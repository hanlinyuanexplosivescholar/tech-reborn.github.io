
### 1.1 `explain()`的执行方法

`explain()`命令可以应用于多种MongoDB操作，包括`find`、`aggregate`、`update`、`delete`等。在`mongosh`（MongoDB Shell）中，主要有两种辅助方法来运行它：`db.collection.explain().<operation>()`和`cursor.explain()`。其中，`db.collection.explain()`方法因其更灵活的链式调用和对各种操作的支持而被认为是首选的执行方式。

### 1.2 详细模式的细致剖析

`explain()`命令接受一个可选的`verbosity`（详细程度）参数，它决定了返回信息的数量和类型。理解这些模式是掌握`explain()`的关键。

#### 1.2.1 `queryPlanner`模式（默认）

这是最轻量级的模式。在此模式下，MongoDB只运行查询优化器来选择优胜计划，但并不会实际执行该计划。它主要返回`queryPlanner`部分的信息，详细说明了优化器选择的计划以及所有被拒绝的候选项。此模式非常适合快速检查一个查询是否使用了预期的索引，例如确定是`IXSCAN`还是`COLLSCAN`。

####1.2.2 `executionStats`模式

该模式是性能分析的首选工具。它会运行查询优化器，选择优胜计划，然后将该计划执行到完成。返回结果中除了包含`queryPlanner`信息外，还会额外提供`executionStats`部分，其中包含了实际执行的统计数据。这些统计数据包括总执行时间、扫描的文档数、扫描的索引键数以及返回的文档数。

#### 1.2.3 `allPlansExecution`模式

这是最详细、资源消耗最高的模式。它不仅会执行优胜计划并返回其完整的执行统计信息，还会在计划选择阶段捕获并返回所有被拒绝的候选计划的部分执行统计。该模式通常被称为“取证模式”。

当`explain("executionStats")`显示一个看似高效的优胜计划（例如，一个低扫描数的`IXSCAN`），但实际查询仍然很慢时，这种模式就变得至关重要。这可能表明在某些特定情况下，查询优化器错误地选择了次优计划。通过使用`allPlansExecution`，分析人员可以检查被拒绝计划的部分统计数据。如果一个被拒绝的计划在计划选择阶段表现出更低的`executionTimeMillis`或`totalDocsExamined`，这可能揭示出优化器在特定数据分布下存在选择缺陷。通过这种方式，`allPlansExecution`模式允许深入分析优化器本身的行为，识别出导致其错误决策的根本原因，从而解决那些难以通过常规诊断发现的性能问题。

下表详细对比了这三种模式：

| 模式名称                | 行为                                               | 输出部分                                                          | 关键用途                                 |
| ------------------------- | ---------------------------------------------------- | ------------------------------------------------------------------- | ------------------------------------------ |
| `queryPlanner`      | 仅分析计划，不执行                                 | `queryPlanner`                                                | 快速检查索引使用、识别优胜和被拒计划     |
| `executionStats`    | 选择并执行优胜计划                                 | `queryPlanner`,`executionStats`                           | 深入性能分析，测量执行时间、扫描数等指标 |
| `allPlansExecution` | 选择并执行优胜计划，同时捕获所有候选计划的统计信息 | `queryPlanner`,`executionStats`（包含所有计划的部分统计） | 复杂性能问题取证，调试优化器决策         |

## 解读`explain()`输出：字段详解

### 2.1 `explain()` JSON文档的结构

`explain()`命令的输出是一个复杂的JSON文档，其核心是一个阶段（stage）树。每个阶段都将它的结果（文档或索引键）传递给父节点。叶子节点通常负责访问集合或索引，而内部节点则处理子节点返回的数据。根节点是最终得出结果集的阶段。需要注意的是，输出格式会因MongoDB版本和查询引擎（经典或基于槽）的不同而有所差异。

### 2.2 `queryPlanner`部分

这一部分详细说明了查询优化器选择的计划，是理解查询路径的基础。

* ​**`winningPlan`**​: 描述了被选为最有效执行路径的文档。
* ​**`rejectedPlans`**​: 一个数组，列出了被优化器考虑但最终因效率低下而被放弃的计划。
* ​**`queryHash`**​: 一个十六进制字符串，代表查询的哈希值，可用于识别具有相同查询形状的慢查询。自MongoDB 8.0起，该字段被`planCacheShapeHash`取代。

### 2.3 `executionStats`部分

这是诊断性能问题的核心数据来源，它提供了查询执行的统计信息。

* ​**`executionTimeMillis`**​: 从计划选择到执行完成的整个过程所花费的总时间，以毫秒为单位。
* ​**`nReturned`**​: 优胜计划实际返回给用户的文档数量。
* ​**`totalDocsExamined`**​: MongoDB为了满足查询，从磁盘中扫描或访问的文档总数。
* ​**`totalKeysExamined`**​: MongoDB为了满足查询，扫描的索引键总数。

单纯看这些数字可能意义有限，但它们的比率关系却能揭示出深刻的性能问题。`totalDocsExamined`与`nReturned`的比值过高，表明数据库为了找到少数匹配的文档而扫描了大量无关文档，这通常是索引缺失或低效的迹象。同样，`totalKeysExamined`与`nReturned`的比值过高，则可能意味着索引设计不当或过滤器选择性低。理想情况下，最有效的查询应使得`totalDocsExamined`和`totalKeysExamined`的值尽可能接近`nReturned`。这表明MongoDB只检查了它真正需要返回的文档和索引键。

下表详细说明了这些关键指标：

| 指标名称                  | 描述                                 | 性能解读                                                         |
| --------------------------- | -------------------------------------- | ------------------------------------------------------------------ |
| `executionTimeMillis` | 查询从计划选择到完成的总耗时         | 值越低越好，高值表明查询性能差                                   |
| `nReturned`           | 查询返回的文档总数                   | 最终结果集大小，用于与扫描数比较                                 |
| `totalDocsExamined`   | 数据库为了执行查询而扫描的文档总数   | 应尽可能接近`nReturned`，值远大于`nReturned`是低效的表现 |
| `totalKeysExamined`   | 数据库为了执行查询而扫描的索引键总数 | 应尽可能接近`nReturned`，值远大于`nReturned`是低效的表现 |

### 2.4 执行阶段：一个分层视图

`explain()`输出的`stage`字段描述了查询计划中的具体操作。理解这些阶段对于诊断至关重要。

| 阶段名称          | 操作描述                                           | 性能影响                                                                           |
| ------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------ |
| `COLLSCAN`    | 全集合扫描。逐个文档扫描整个集合以查找匹配项。     | 成本最高的非优化操作，通常是缺少索引的标志。                                       |
| `IXSCAN`      | 索引扫描。使用索引键来查找匹配的文档。             | 高效操作，是大多数查询优化的首要目标。                                             |
| `FETCH`       | 从集合中检索实际文档。通常在`IXSCAN`之后发生。 | 从磁盘读取文档，效率通常取决于`IXSCAN`的结果集大小。                           |
| `SORT`        | 在内存中对文档进行排序。                           | 如果无法使用索引进行排序，则会触发此阶段。它会占用大量内存，并且可能导致查询失败。 |
| `PROJECTION`  | 过滤返回字段。                                     | 减少网络和内存开销，对于实现覆盖查询至关重要。                                     |
| `SHARD_MERGE` | 合并来自不同分片的结果集。                         | 在分片集群中，这是查询执行的最后一个阶段。                                         |

## 使用`explain()`进行性能调优的实践指南

### 3.1 案例研究1：诊断一个缓慢的`find()`查询

#### 3.1.1 步骤1：运行`explain("executionStats")`

当怀疑一个查询性能低下时，第一步是使用`executionStats`模式来运行`explain()`。这会提供完整的执行统计数据，而不仅仅是计划本身。

#### 3.1.2 步骤2：识别`COLLSCAN`及其成因

分析`explain()`输出，检查`winningPlan.stage`字段。如果其值为`COLLSCAN`，则表明查询没有使用索引，而是扫描了整个集合。同时，`totalDocsExamined`的值会接近或等于集合中的文档总数，而`totalKeysExamined`将为0，这些都是全集合扫描的明确信号。

#### 3.1.3 步骤3：创建正确的索引

一旦确定了`COLLSCAN`是瓶颈，解决方案就是为查询字段创建索引。例如，`db.collection.createIndex({ "field_name": 1 })`。创建索引后，再次运行`explain()`来验证查询计划。此时，`winningPlan`应显示`IXSCAN`阶段作为其`inputStage`，并且`totalDocsExamined`和`totalKeysExamined`的值将显著减少，接近于`nReturned`的值。

### 3.2 案例研究2：优化排序操作

#### 3.2.1 识别内存中的排序

如果`explain()`输出中包含一个独立的`SORT`阶段，则意味着MongoDB无法使用索引来满足排序需求，而是被迫在内存中对文档进行排序。这种操作是低效的，并且存在100MB的内存限制，一旦超出将导致查询失败。

#### 3.2.2 建立复合索引以实现高效排序

为了消除`SORT`阶段，需要创建一个复合索引，使其能够同时支持查询的过滤和排序。在设计复合索引时，有一个重要的经验法则：ESR（Equality, Sort, Range）规则。这个规则建议将字段按以下顺序排列在复合索引中：

1. ​**Equality (等值)**​：首先放置用于精确匹配（如`$eq`）的字段。
2. ​**Sort (排序)**​：其次放置用于排序的字段。
3. ​**Range (范围)**​：最后放置用于范围查询（如`$gt`, `$lte`）的字段。

遵循这个规则可以创建一个单一的、连贯的索引结构，允许查询优化器使用一个`IXSCAN`操作同时完成文档的过滤和排序，从而消除成本高昂的`SORT`阶段。

### 3.3 案例研究3：实现覆盖查询

#### 3.3.1 什么是覆盖查询？

覆盖查询是一种特殊的高效查询，其返回的所有字段都包含在索引中，因此MongoDB无需访问实际的文档即可直接从索引中返回结果。

#### 3.3.2 如何使用`explain()`确认覆盖查询

要确定查询是否为覆盖查询，可以运行`explain()`并检查`executionStats.totalDocsExamined`字段。如果该值为0，则表示查询被索引完全覆盖。在`winningPlan`中，通常只会看到`IXSCAN`和`PROJECTION`阶段，而不会出现`FETCH`阶段，因为不需要从集合中获取文档。

#### 3.3.3 覆盖查询的最佳实践

在设计覆盖查询时，一个常见的陷阱是`_id`字段。MongoDB默认总是返回`_id`，因此如果`_id`字段没有包含在索引中，即使所有其他字段都在索引中，查询也无法被完全覆盖。为了解决这个问题，需要显式地从返回结果中排除`_id`，或将其作为复合索引的一部分。对于分片集群，分片键也必须作为索引的一部分，才能实现覆盖查询。

## 高级场景：`explain()`在复杂操作中的应用

### 4.1 `explain()`在写入操作中的应用

`explain()`命令不仅适用于读取操作，也可以应用于`update`、`delete`和`findAndModify`等写入操作。一个重要的安全机制是，当对写入操作执行`explain()`时，它只会返回将要执行的计划信息，但**不会**实际修改数据库中的数据。这使得它成为一个安全的测试和分析写入操作性能的工具。

### 4.2 `explain()`与聚合管道

对于复杂的聚合管道，`explain()`同样是不可或缺的工具。它可以应用于整个聚合管道，详细展示数据流经每个阶段（如`$match`, `$group`, `$sort`等）时的转换和性能表现。然而，需要注意的是，`explain()`在`executionStats`或`allPlansExecution`模式下运行时，不能与包含`$out`阶段的聚合管道一起使用，因为它会试图修改数据库。在这种情况下，分析人员可以选择在`queryPlanner`模式下运行`explain()`，或者在不包含`$out`阶段的情况下运行，以分析管道的前几个阶段。

### 4.3 分片环境下的`explain()`

在分片集群中，`explain()`输出会包含一个`shards`字段。这个字段下会列出每个被查询访问的分片，并为每个分片提供独立的`explain`输出。这使得开发人员可以逐个分析每个分片上的查询计划，并识别在特定分片上可能存在的性能问题。
